# 推广引流工作流

## 目标

在他人发布的**相关**视频下做自然引流，导向主推作品。

## 前置条件（必须全部完成）

```
□ 已加载 用户配置.md
□ 已加载 全局规则.md（含评论筛选过滤规则）
□ Bridge Server 在线：bash ~/.claude/skills/douyin-cli/scripts/bridge.sh ensure
□ 油猴脚本已连接：node cli.js status
□ REPLIED_CIDS 集合已初始化（使用执行模板.md中的 init_replied_cids 函数）
□ 已确定推广目标（话题/作者）
```

## 🔴 核心安全规则（必须严格遵守）

### 规则 1：命令串行 + 强制随机间隔

```
┌─────────────────────────────────────────────────────────────┐
│  ❌ 严禁并发执行任何 CLI 命令                                  │
│  ❌ 严禁使用固定间隔（必须每次重新随机）                         │
│  ❌ 严禁跳过 sleep（即使"感觉"不需要）                          │
│                                                              │
│  ✅ 所有命令必须逐条串行执行                                   │
│  ✅ 写操作每条命令完成后必须等待 40-55 秒随机间隔                │
│  ✅ 读操作每条命令完成后必须等待 30-50 秒随机间隔                │
│  ✅ 每次等待前必须重新生成随机数                                │
└─────────────────────────────────────────────────────────────┘
```

**强制执行模板**（复制使用，不要修改结构）：

```bash
# ═══ 执行单条命令的模板 ═══
execute_with_delay() {
  local cmd="$1"
  local type="${2:-write}"  # write 或 read
  local delay
  
  if [ "$type" = "read" ]; then
    delay=$((30 + RANDOM % 21))  # 30-50 秒随机值
  else
    delay=$((40 + RANDOM % 16))  # 40-55 秒随机值
  fi
  
  echo "▶ 执行: $cmd"
  eval "$cmd"
  
  if [ $? -eq 0 ]; then
    echo "✓ 成功，等待 ${delay} 秒..."
    sleep "$delay"
  else
    echo "✗ 失败，跳过等待"
  fi
}
```

### 规则 2：不重复回复

```
┌─────────────────────────────────────────────────────────────┐
│  ❌ 同一条评论 cid 一生只能被回复一次（跨日跨轮均生效）           │
│  ❌ 同一作者短期内不重复（≥ 7 天冷却期）                        │
│                                                              │
│  ✅ 每次回复前必须检查 cid 是否在 REPLIED_CIDS 中              │
│  ✅ 回复成功后必须将 cid 加入 REPLIED_CIDS                    │
│  ✅ 必须检查同一作者的最后互动时间（≥ 7 天）                    │
└─────────────────────────────────────────────────────────────┘
```

**强制执行流程**：

```bash
# ═══ 检查是否已回复 ═══
is_replied() {
  local cid="$1"
  grep -q "$cid" /tmp/replied_cids.txt 2>/dev/null
  return $?
}

# ═══ 检查作者冷却期（≥ 7 天）═══
check_author_cooldown() {
  local user_id="$1"
  local last_interaction=$(node cli.js profile "$user_id" | grep -o '"last_seen":"[^"]*"' | cut -d'"' -f4)
  
  if [ -z "$last_interaction" ]; then
    return 0  # 无记录，可以互动
  fi
  
  local last_ts=$(date -d "$last_interaction" +%s 2>/dev/null || echo 0)
  local now_ts=$(date +%s)
  local days_diff=$(( (now_ts - last_ts) / 86400 ))
  
  if [ $days_diff -ge 7 ]; then
    return 0  # 冷却期已过
  else
    echo "⏭ 跳过：作者 $user_id 冷却期未过（${days_diff}/7 天）"
    return 1  # 冷却期未过
  fi
}

# ═══ 回复前检查 ═══
if is_replied "target_cid"; then
  echo "⏭ 跳过：已回复过该评论"
elif ! check_author_cooldown "target_user_id"; then
  echo "⏭ 跳过：作者冷却期未过"
else
  execute_with_delay 'node cli.js post <aweme_id> "内容" --reply-to target_cid'
  echo "target_cid" >> /tmp/replied_cids.txt  # 记录已回复
fi
```

## 视频筛选标准（推广场景专用）

> 推广场景比互动场景更严格，不仅筛选评论，还要筛选**视频本身**。

### 跳过这些视频

- 纯广告/营销内容
- 与推广目标相关度低
- 评论区冷清（< 5 条评论）
- 发布时间过久（> 30 天）

### 优先选择

- 目标作者的视频
- 讨论技术/方案的视频
- 评论区有深度讨论的视频

## 评论筛选标准

### 通用规则

遵循 `全局规则.md` 中的评论筛选过滤规则

### 推广场景额外跳过

- 已有其他人推广过的评论区
- 评论区氛围偏负面/攻击性
- 视频作者明显是竞品

### 推广场景优先回复

- 视频作者本人的评论（最高优先级）
- 提出与推广内容相关问题的评论
- 表达兴趣/需求的评论

## 相关性分析（必做前置步骤）

❗ 不要直接套用固定关键词列表，要从"主推作品"反推。

**Step 1 — 提取核心主题词（3–5 个）**

从主推作品出发，提取：
- **技术维度**（如 "CDP / 油猴 / 浏览器自动化 / Node CLI"）
- **场景维度**（如 "评论自动回复 / 评论分析 / 私信批量"）
- **受众维度**（如 "自媒体运营 / 程序员副业 / 内容创作者"）
- **价值维度**（如 "效率工具 / 开源 / 零依赖"）

将这些写在执行日志里，作为本轮"相关性锚点"。

**Step 2 — 派生搜索词**

每个主题词派生 2–3 个搜索变体（同义词 / 相邻话题），形成搜索队列：

```
node cli.js search "<主题派生词>" --count 5
```

**Step 3 — 候选视频相关度评分**

对每个候选视频按下面打分（满分 5，≥ 3 才进入下一步）：

| 维度 | 1 分 | 0 分 |
|------|------|------|
| 视频 desc 是否提到主题词 / 同义词 | 是 | 否 |
| 评论数 < 100 | 是 | 否 |
| 近期活跃 (`time` 在最近 30 天内) | 是 | 否 |
| 不需要"先解释项目是啥"就能自然引出 | 是 | 否 |
| 同作者最近 7 天没被自己评过 (查 `audit.json`) | 是 | 否 |

❗ 任意一个 0 分等于全局禁令命中（§3 / §2.1）→ **跳过**，记录原因。

## 候选来源（两路并行）

通过相关度评分的视频，**两路都要走**（单轮内 ≈ 1:1）：

1. **直接评论视频本身** — 顶层评论。
2. **回复评论区里的高相关评论** — 对评论区中表达过相关需求 / 提过相关问题的评论用 `--reply-to` 接住。

## 风格轮换（每轮 ≥ 3 种风格，参考 §4 内容禁令）

- 🗣️ **自然聊天** — "我也做了个 Yht20927/douyin-cli，最近刚加了 xxx 功能..."
- 🤔 **提问式** — "有人试过用 CDP 做评论自动化吗？我搞了个 Yht20927/douyin-cli..."
- 😂 **自嘲式** — "打工人给自己写了 Yht20927/douyin-cli，省了每天 2 小时回评论..."
- 💡 **技术对比** — "Yht20927/douyin-cli 用 CDP 方案比 xxx 更轻量，star 已经 xxx 了..."
- 🙏 **价值分享** — "Yht20927/douyin-cli 免费开源的，最近刚发布了 v2.x，欢迎交流"

> ⚠️ 以上仅为风格示意，实际推广内容必须基于 gh-cli 获取的**真实仓库信息**生成，禁止照搬模板。

### Step 2: 提取主题词 → 派生搜索词

按"相关性分析"章节执行


评分 < 3 → 跳过，记录原因

a. **直接评论**：

## 专属限制

- 每轮总条数（直接评论 + 回复评论） ≤ **10**。
- 直接评论 ∶ 回复评论 ≈ **1∶1**，不押单边。
- 单条间隔 ≥ **1 分钟**。
- 每日 ≤ **3** 轮（合计 ≤ 30 条），超出 2 轮后逐轮观察风控信号，详见全局规则.md §1。
- ⚠️ §2.1 / §3 / §4 全局禁令同时生效。

## 推广策略详解

### 策略 A：解答引导型

**适用场景**：评论中有提出问题

**示例**：
> 评论："这个方案的性能瓶颈在哪？"
> 回复："这个问题其实可以用 [方案] 解决，我之前研究过，效果不错。关键是 [要点]。"

**要点**：
- 直接回答问题
- 提供具体方案
- 暗示有更多内容

### 策略 B：共鸣分享型

**适用场景**：评论中有分享经验或表达困惑

**示例**：
> 评论："我之前也遇到过这个问题，折腾了好久"
> 回复："深有同感！我之前也遇到过，后来发现 [经验]，分享给大家参考。"

**要点**：
- 表达共鸣
- 分享相关经验
- 提供实用建议

### 策略 C：补充延伸型

**适用场景**：评论中有讨论但可以补充更多角度

**示例**：
> 评论："这个方案不错，但成本有点高"
> 回复："说得对！补充一点，[额外信息] 这个也很重要，大家可以了解一下。"

**要点**：
- 认同原观点
- 补充有价值信息
- 引导进一步讨论

## 增量执行

后续执行时使用增量模式：

```bash
# ═══ 搜索同一关键词，只关注新视频（使用执行模板）═══
execute_with_delay 'node cli.js search "AI Agent" --count 30' read
# 筛选发布时间在上次执行之后的视频

# ═══ 获取视频新评论（使用执行模板，优先使用增量模式）═══
execute_with_delay 'node cli.js get <aweme_id> --new --depth 1' read
```

**增量策略**：
- 话题推广：定期搜索同一关键词，关注新视频
- 作者推广：关注目标作者的新视频
- 评论去重：依赖 REPLIED_CIDS

## 注意事项

1. **自然融入**：不要让推广感太强，像正常交流
2. **提供价值**：每条评论都要有实质内容
3. **遵守冷却期**：同一作者 ≥ 7 天（使用 check_author_cooldown 函数）
4. **避免竞品贴脸**：不在竞品内容下推广
5. **评论筛选**：遵循 `全局规则.md` 第 7 节 + 推广场景额外规则
6. **间隔执行**：所有命令必须使用 `execute_with_delay` 模板
7. **增量获取**：优先使用 `--new` 模式获取评论
8. **真实数据**：推广评论必须包含 gh-cli 获取的真实仓库信息

