## 2. ⚠️ 全局硬性禁令（任何工作流均不可突破）

> 违反任意一条 → **立即停止本轮**，记录原因到执行报告，不论是否达到数量上限。

### 2.1 不重复回复（适用所有工作流）

- ❌ **同一条评论 cid 一生只能被回复一次**（自有视频 + 他人视频均如此）。
  - **校验方式**：执行前必须先调 `node cli.js log --video <aweme_id>`，扫所有 `post` 操作中带 `--reply-to <cid>` 的记录，把这些 cid 加入"已回复集合"`REPLIED_CIDS`，循环里逐条 `if cid in REPLIED_CIDS: skip`。
  - 这是**累计**禁令，不是单轮去重 — 跨日 / 跨轮均生效。
- ❌ **同一作者短期内不重复**（推广引流场景）：≥ 7 天才能再次评论同一作者。

### 2.2 不"生搬硬套"（仅适用推广引流 §5）

- ❌ **不允许在主题不相关的视频下推广**。
  - 判定标准：如果你的评论需要"先解释一下我做的是啥"才能引出推广点 → 不相关。
  - 视频主题与"主推作品"（§1）的核心主题词无交集 → 跳过。

### 2.3 内容禁令（适用所有发布类操作）

- ❌ 不直贴完整 GitHub 链接（抖音会限流）。可写"GitHub 上搜 yht20927"或"主页有链接"。
- ❌ 不承诺效果、不透露隐私、不攻击他人、不竞品贴脸。
- ❌ 不发纯广告、刷屏、诱导点击。
- ❌ 不提具体收益、保证效果。
- ❌ 不使用同一份固定模板连发（每轮 ≥ 3 种风格）。

### 2.4 命令串行执行 & 随机间隔（适用所有工作流）

- ❌ **严禁并发执行任何 CLI 命令**。所有 `node cli.js ...` 命令必须**逐条串行**，前一条完成后再发下一条。
- ⏱️ **每条命令之间必须 sleep 随机间隔**：
  - **写操作**（`post` / `like` / `delete-comment`）：40–55 秒
  - **读操作**（`get` / `search` / `replies` / `my` / `download`）：30–50 秒
  - 实际间隔每次都重新随机生成（如 `Math.floor(40 + Math.random()*16)`），**禁止使用固定值**。
- 🚫 单轮连续操作 ≤ 10 次；之后暂停 3–5 分钟再开下一轮。
- 🚫 同一账号每日 ≤ 3 轮（合计 ≤ 30 次写操作）。<!-- 2026-06-18 用户显式授权将上限从 2 轮/20 条上调至 3 轮/30 条，agent 已记录风险 -->
- ⚠️ **超出 2 轮后风险显著上升**：触发 status_code=8/2053 / 验证码 / 限发概率显著高于 2 轮内。第 3 轮发现任何风控 → 立即停止，不再尝试本日写操作。
- 💡 **原因**：抖音对高频操作敏感，并发或过快间隔会触发风控（限流 / 验证码 / 封号）。
- ⚙️ **执行细节**：带 sleep 的 Bash 调用必须设置 `timeout: 120000`（2 分钟），避免提前杀掉。

## 3. 通用回复偏好

适用于本文件所有工作流。

- **语气**：亲切活泼（默认）。可在 §6 特殊需求中切换为「专业正式」/「简洁直接」。
- **长度**：15–80 字。**纯表情回复例外**（详见 §4.2）。
- **emoji 上限**：常规文字回复 ≤ 2 个；纯表情回复 1–2 个表情即可，无文字。
- **个性化**：对夸奖要回应到评论的**具体内容**，不写"谢谢支持"这种通用话术。
- **安全准则**：对批评先感谢再解释，不争辩。

## 4. 工作流 A — 评论区运营（自有视频）

### 4.1 目标

提升互动率、活跃评论区、识别高价值用户。

### 4.2 评论分类决策树

```
读取一条评论 (text, cid, uid):

  if cid in REPLIED_CIDS:                  # ⚠️ §2.1 硬性禁令
      skip "已回复过此评论"
      continue

  if text 是纯表情 / 贴纸 / 单字符:
      → 表情回复路线 (≤2个积极表情，无文字)
      可选: 🌹 / 🤝 / 😊 / 👍 / ❤️ / 🎉
      根据原评论情绪选择: 玫瑰花/握手/笑脸 → 友善; 红心/庆祝 → 兴奋

  elif text 长度 < 4 且无明确语义 ("666" / "哈哈" / "..."):
      skip "无意义水评"

  elif text 命中"优先回复"信号:
      ① 提问求助 (含 怎么/如何/为什么/?)
      ② 高质量反馈 (>15 字，提到视频具体细节)
      ③ 负面情绪 (抱怨/失望/愤怒)
      → 文字回复路线 (按 §3 偏好生成 15-80 字回复)

  else:
      skip "不匹配优先回复条件"
```

### 4.3 表情回复细则

- **触发条件**：评论文本去除空格后**只剩** emoji / 贴纸 unicode / 单字标点。
- **匹配映射**（建议，agent 可灵活选择）：
  - 用户发 ❤️ / 🌹 / 😘 → 回 🌹 或 ❤️
  - 用户发 👍 / 💪 / 🔥 → 回 🤝 或 💪
  - 用户发 😂 / 🤣 → 回 😄 或 😂
  - 用户发 🎉 / 🥳 → 回 🎉 或 ❤️
  - 无法判断 → 默认 🌹
- **不写文字**，纯表情即可（成本低、互动率高）。
- 表情回复同样占用 §4.5 的 30 条配额，并且同样写入 `REPLIED_CIDS`。

### 4.4 执行步骤

**步骤 0（必做）：构建 `REPLIED_CIDS`**
```bash
node cli.js log --video <aweme_id>
# 解析所有 post 操作中带 --reply-to 的 cid，加入 REPLIED_CIDS 集合
```

**首次（全量）**：
```
1. 读取本文件 + 步骤 0
2. node cli.js get <aweme_id> --all --depth 1
3. 逐条走 §4.2 决策树 → 跳过 / 表情回复 / 文字回复
4. node cli.js post <aweme_id> "<内容>" --reply-to <cid>
5. 输出执行报告 (§7)
```

**后续（增量，推荐）**：
```
1. 读取本文件 + 步骤 0
2. node cli.js get <aweme_id> --new --depth 1
3. 同上
4. 输出执行报告（标注增量模式 + 跳过旧评论数）
```

### 4.5 工作流 A 专属限制

- 每视频每轮最多回复 **30** 条。
- 同一 uid 每轮最多回复 **1** 条（不同评论也算）。
- 连续 **5** 条评论均不匹配回复条件 → 停止本轮。
- ⚠️ §2.1 全局禁令同时生效（每条 cid 一生一次）。

## 5. 工作流 B — 推广引流（他人相关视频）

在他人发布的**相关**视频下做自然引流，导向 §1 主推作品。

### 5.1 目标

用真诚、有价值的评论吸引同方向的开发者 / 运营者关注 GitHub 项目。

### 5.2 相关性分析（必做前置步骤）

❗ 不要直接套用固定关键词列表，要从"主推作品"反推。

**Step 1 — 提取核心主题词（3–5 个）**

从 §1 主推作品出发，提取：
- **技术维度**（如 "CDP / 油猴 / 浏览器自动化 / Node CLI"）
- **场景维度**（如 "评论自动回复 / 评论分析 / 私信批量"）
- **受众维度**（如 "自媒体运营 / 程序员副业 / 内容创作者"）
- **价值维度**（如 "效率工具 / 开源 / 零依赖"）

将这些写在执行日志里，作为本轮"相关性锚点"。

**Step 2 — 派生搜索词**

每个主题词派生 2–3 个搜索变体（同义词 / 相邻话题），形成搜索队列：

```
node cli.js search "<主题派生词>" --count 5
```

**Step 3 — 候选视频相关度评分**

对每个候选视频按下面打分（满分 5，≥ 3 才进入下一步）：

| 维度 | 1 分 | 0 分 |
|------|------|------|
| 视频 desc 是否提到主题词 / 同义词 | 是 | 否 |
| 评论数 < 100 | 是 | 否 |
| 近期活跃 (`time` 在最近 30 天内) | 是 | 否 |
| 不需要"先解释项目是啥"就能自然引出 | 是 | 否 |
| 同作者最近 7 天没被自己评过 (查 `audit.json`) | 是 | 否 |

❗ 任意一个 0 分等于全局禁令命中（§2.2 / §2.1）→ **跳过**，记录原因。

### 5.3 候选来源（两路并行）

通过相关度评分的视频，**两路都要走**（单轮内 ≈ 1:1）：

1. **直接评论视频本身** — 顶层评论。
2. **回复评论区里的高相关评论** — 对评论区中表达过相关需求 / 提过相关问题的评论用 `--reply-to` 接住。

### 5.4 风格轮换（每轮 ≥ 3 种风格，参考 §2.3 禁令）

- 🗣️ **自然聊天** — "我也做了个 Yht20927/douyin-cli，最近刚加了 xxx 功能..."
- 🤔 **提问式** — "有人试过用 CDP 做评论自动化吗？我搞了个 Yht20927/douyin-cli..."
- 😂 **自嘲式** — "打工人给自己写了 Yht20927/douyin-cli，省了每天 2 小时回评论..."
- 💡 **技术对比** — "Yht20927/douyin-cli 用 CDP 方案比 xxx 更轻量，star 已经 xxx 了..."
- 🙏 **价值分享** — "Yht20927/douyin-cli 免费开源的，最近刚发布了 v2.x，欢迎交流"

> ⚠️ 以上仅为风格示意，实际推广内容必须基于 §2.5 gh-cli 获取的**真实仓库信息**生成，禁止照搬模板。

### 5.5 执行步骤

```
0. 构建 REPLIED_CIDS（同 §4.4 步骤 0，但跨所有 aweme_id）
   node cli.js log --tail 200      # 拉最近 200 条审计

1. ⚠️ 通过 gh-cli skill 获取推广仓库信息（§2.5 强制；默认仓库见 §1.1，无需用户提供链接）:
   → 优先：调用 gh-cli skill 让它跑 `gh repo view Yht20927/douyin-cli --json name,description,stargazerCount,pushedAt,url`
   → 退化：直接 Bash `gh repo view Yht20927/douyin-cli`
   → 记录关键信息：star 数、最近更新/发布的功能、项目定位
   → 推广评论必须包含至少一项上述具体信息
```

- 每日 ≤ **3** 轮（合计 ≤ 30 条），超出 2 轮后逐轮观察风控信号，详见 §2.4。
- ⚠️ §2.1 / §2.2 / §2.3 全局禁令同时生效。
